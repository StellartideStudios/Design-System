# Product

## Register

product

## Users

Stellartide Studios builders use this design system while creating apps and websites across Expo, React Native, TypeScript, and Next.js. They need reusable primitives, typed theme objects, and component guidance that can move between native app surfaces and web surfaces without losing craft.

The immediate user is a developer-designer shaping a reusable foundation before product screens exist. Future users include collaborators who need clear token rules, accessible component behavior, and a visual language that can support both calm product workflows and high-energy brand moments.

## Product Purpose

The design system exists to make Stellartide interfaces consistent, reusable, and expressive across platforms. It should provide durable tokens, a light/dark theme model, core primitives, first components, and a gallery surface that behaves like a tiny internal Storybook without adding Storybook yet.

Success means new app and web screens can be built from the system without re-deciding color roles, spacing, typography, radius, elevation, motion, or basic interaction states each time.

## Brand Personality

High-energy, graphic, angular, and polished.

The system should carry the punch of a game-event campaign reference while keeping the day-to-day product layer practical. It can be bold at the edges, electric in accents, and sharp in composition, but the reusable product foundation must remain readable, accessible, and steady.

## Anti-references

- Generic SaaS dashboard sameness.
- Beige, muted, over-safe design systems with no point of view.
- Direct clones of HoYoverse, Genshin Impact, character art, event logos, or campaign assets.
- Decorative effects that overwhelm product usability.
- Dark mode with purple gradients as a reflex.
- Components that look exciting once but become tiring in repeated app workflows.

## Design Principles

1. Build the quiet core before the loud edge. Product primitives should be calm enough for daily use, while brand and campaign variants can carry the electricity.
2. Make tokens reusable before they are dramatic. Color, spacing, type, radius, shadow, z-index, and motion must support real screens, not only posters.
3. Treat accessibility as a design material. Focus, disabled, pressed, contrast, and reduced-motion states are part of the visual language.
4. Keep cross-platform behavior honest. Expo, React Native, and Next.js should share intent even when implementation details differ.
5. Use the reference energy, not the reference IP. The goal is Stellartide's own sharp, saturated, game-event polish.

## Accessibility & Inclusion

Target WCAG AA contrast for text and interactive states. Components should support screen-reader roles and labels, keyboard focus where supported, disabled states, pressed states, and reduced-motion preferences. Color must not be the only signal for selection, success, warning, danger, or focus.
